## 🎯 프로젝트 개요

### 서비스명
**뷰티 AI 어드바이저** (Beauty AI Advisor)

### 목적
화장품 리뷰 분석 서비스의 프로토타입 구현. 30,000+ 건의 리뷰 데이터를 시각화하여 소비자가 제품을 빠르게 비교하고 선택할 수 있도록 지원.

**현재 단계 (merged)**:
- `merged` 브랜치: `merge_first` 기반으로 생성된 통합 브랜치 (frontend + backend 병합 상태)
- `origin/backend` 브랜치의 모든 변경사항이 포함됨 (FastAPI MVP + SQLite DB 기반)
- 프론트엔드가 실제 백엔드 DB 데이터 기반으로 동작 (`VITE_USE_MOCK=false`)
- `origin/agent` 브랜치 발견됨 (구버전 지식동기화 문서만 포함, 백엔드 코드 없음 — 참고용)
- LLM AI Agent는 아직 미구현 (확장 가능성만 고려)
- /api/chat, /api/compare 엔드포인트는 미구현 (챗봇/비교 페이지는 임시 mock 유지)

### 핵심 기능 (v1.0 프로토타입)
1. **대시보드 홈**: KPI 카드 (총 리뷰수, 평균 별점, 긍정/부정 비율)
2. **시계열 차트**: 긍정/부정 리뷰 추이 시각화
3. **키워드 분석**: 긍정/부정 키워드 Top 10 리스트
4. **제품 필터**: 드롭다운으로 제품 선택

## 🤝 협업 프로토콜

### 1. API 계약 (Contract)

프론트엔드와 백엔드는 다음 API 계약을 준수합니다:

#### API 엔드포인트 목록 (구현 완료)

| 엔드포인트 | 메서드 | 설명 | 쿼리 파라미터 |
|-----------|--------|------|------|
| `/api/products` | GET | 전체 제품 목록 | - |
| `/api/kpi` | GET | KPI 데이터 | `product_code` 또는 `product` |
| `/api/timeseries` | GET | 시계열 데이터 | `product_code` 또는 `product` |
| `/api/keywords/positive` | GET | 긍정 키워드 | `product_code`, `limit` |
| `/api/keywords/negative` | GET | 부정 키워드 | `product_code`, `limit` |
| `/api/categories` | GET | 카테고리별 언급 수 | `product_code` 또는 `product` |
| `/api/category/reviews` | GET | 카테고리별 리뷰 | `product_code`, `major`, `minor`, `sentiment` |

**미구현 (v2.0 예정)**:
| `/api/chat` | POST | LLM 챗봇 응답 | - |
| `/api/compare` | POST | AI 기반 제품 비교 | - |

**응답 형식**: JSON (상세 스키마는 백엔드 문서 참조)

**에러 처리**:
- 200: 성공
- 404: 제품명 없음
- 500: 서버 오류

---

### 2. 변경사항 동기화 규칙

#### API 계약 변경 시
1. **Backend Agent**가 API 변경
2. 백엔드 문서의 "API 엔드포인트 명세" 섹션 업데이트
3. 프론트엔드 문서의 "API 계약서" 섹션 업데이트
4. **이 문서**의 "변경 이력" 섹션에 기록
5. **Frontend Agent**가 코드 수정
6. **QA Agent**가 재테스트

#### 기능 추가 시
1. **이 문서**에 기능 추가 계획 기록 ("작업 현황" 섹션)
2. Frontend/Backend Agent가 각자 문서에 태스크 추가
3. 구현 완료 후 "완료" 상태로 변경
4. QA Agent가 테스트 체크리스트에 항목 추가
5. 테스트 통과 후 이 문서에 "배포 가능" 표시

---

### 3. 버그 리포트 흐름

```
[발견] → [기록] → [할당] → [수정] → [재테스트] → [종료]
  ↓        ↓         ↓        ↓          ↓          ↓
QA      디버깅문서  담당Agent 담당Agent   QA        상태변경
```

**상세 절차**:
1. **QA Agent**가 버그 발견
2. **디버깅 문서**에 버그 로그 기록 (ID, 심각도, 재현 시나리오)
3. **이 문서**의 "현재 이슈" 섹션에 추가
4. 해당 **Agent (Frontend/Backend)**가 수정
5. 수정 완료 후 해당 문서의 "변경 이력" 업데이트
6. **QA Agent**가 재테스트 및 버그 상태 "Closed"로 변경

## 🚀 프로젝트 실행 가이드

### Backend 실행

```bash
# 1. 백엔드 디렉토리로 이동
cd sillim_custom_project_1/backend

# 2. 가상 환경 생성 (최초 1회)
python -m venv venv

# 3. 가상 환경 활성화 (Windows)
venv\Scripts\activate

# 4. 의존성 설치 (최초 1회)
pip install -r requirements.txt

# 5. DB 초기화 (최초 1회 또는 데이터 재적재 시) - reviews.db가 이미 있으면 생략 가능
python init_db.py

# 6. 서버 실행
uvicorn app.main:app --reload
```

**접속**: http://localhost:8000
**API 문서**: http://localhost:8000/docs

> **주의**: `uvicorn`은 반드시 `backend/` 디렉토리에서 실행해야 합니다 (`reviews.db` 경로가 상대 경로).

## 📌 주의사항 및 가이드라인

### 1. 에이전트 경계 준수
- 각 에이전트는 자신의 **작업 범위**만 수정
- 다른 영역 수정 필요 시 → 해당 에이전트에게 요청 (문서에 기록)

### 2. 문서 동기화 필수
- 코드 변경 시 → 반드시 해당 문서 업데이트
- API 계약 변경 시 → 양쪽 문서 모두 업데이트

### 3. 버그는 즉시 기록
- 버그 발견 즉시 → 디버깅 문서에 기록
- 재현 시나리오 명확히 작성

### 4. 독립적 작업
- 각 에이전트는 **독립 세션**에서 작업
- 이 문서 및 담당 문서만 참조하여 작업 가능

### 5. 프로토타입 우선
- 완벽한 구현보다 **빠른 프로토타입**
- LLM AI Agent 기능은 v2.0에서 추가 예정
